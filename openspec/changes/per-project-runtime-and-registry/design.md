# Design: Per-project runtime state, global registry, and a Bun-backed CLI-first local workflow

## Summary

DELIVERATOR will become a global control surface over many registered projects instead of a single-project app bound to the DELIVERATOR repo root. Every managed project gets its own `.deliverator/shared` and `.deliverator/local` tree. The application keeps its registry database and app-level preferences under `~/.deliverator`, but project state lives with each project. The local runtime becomes CLI-first and no longer depends on Docker or a separate observability backend. Bun becomes the package manager and primary command runner for the workspace.

The runtime shape after this change is:
- global application state under `~/.deliverator`
- project-shared config under `<project>/.deliverator/shared`
- project-local runtime state under `<project>/.deliverator/local`
- a foreground CLI command, `deliverator start`, that boots the server and records runtime metadata
- a lightweight CLI helper, `deliverator open`, that opens the current running instance
- searchable JSONL logs instead of a local telemetry stack
- a Bun-backed contributor watch loop exposed through `bun run dev`
- project-scoped board/task routes and APIs
- global dashboard/feed/SSE that aggregate across registered project databases

## Architectural Constraints

- The change is a breaking cutover from the old single-project repo-root runtime model.
- `docs/research/` remains reference-only.
- Fastify remains the single application server for API, SSE, and site hosting.
- Dashboard and feed are global, but board and task detail are project-scoped.
- Global aggregation uses request-time fan-out across project databases in this phase. No central event mirror is introduced.
- `.deliverator/shared` is versionable by design; `.deliverator/local` is local-only by design.
- Docker is not part of the supported runtime architecture.
- OpenTelemetry, Prometheus, Grafana, Loki, Tempo, and Promtail are removed from the supported local runtime architecture.
- Bun replaces `pnpm` and `make dev` in the supported contributor workflow.
- The current app/runtime still uses a compatible Node execution path underneath `bun run ...` because direct Bun runtime remains incompatible with key dependencies in this phase.

## Storage Ownership

### Global application state

The DELIVERATOR application keeps a registry database and app-level state under `~/.deliverator`. This storage contains only:
- registered project records
- app state such as the last selected project
- app-instance operational metadata if needed later

It does not store tasks, runs, comments, artifacts, or workflow definitions for managed projects.

The initial contract for this storage is:
- `~/.deliverator/data/registry.db` for registry and app-state tables
- `~/.deliverator/run/current.json` for the currently running local instance metadata
- `~/.deliverator/logs/` for application-global JSONL logs if they are emitted outside a project context
- future global caches or indexes only if they are truly cross-project

### Project-shared state

Each managed project gets `<project>/.deliverator/shared`. This directory contains the config that should survive across machines and be shareable through version control:
- `workflow.yaml`
- `project.yaml`
- `recipes/`
- `prompts/`
- `schemas/`
- `validators/`

The current default product files generated by `packages/core/src/init.ts` move here.

### Project-local state

Each managed project gets `<project>/.deliverator/local`. This directory contains:
- `deliverator.db`
- `artifacts/`
- `worktrees/`
- `logs/`

This is the local source of truth for that project’s tasks, runs, task events, comments, approvals, and evidence.

Project-local logs also live here so `deliverator logs` can search either one project or all projects without a log shipping service.

## Registry and Project Contexts

### Registry database

Add a small global-registry schema with tables such as:
- `registered_projects`
- `app_state`

`registered_projects` stores at least `id`, `slug`, `name`, and `root_path`. `app_state` stores singleton preferences such as `last_selected_project_slug`. These tables live in `~/.deliverator/data/registry.db`.

### Project context manager

The current startup path opens one database and one workflow. Replace that with a `ProjectContextManager` that can:
- look up a registered project by slug
- resolve its `<project>/.deliverator/shared` and `local` paths
- ensure required directories exist
- open and migrate the project-local database
- compile the project’s workflow from `.deliverator/shared/workflow.yaml`
- cache the resulting context in memory

This allows project-scoped routes to work without treating the app repo root as the only project root.

## Runtime and API Boundaries

### CLI runtime

Add a new executable app package, `apps/cli`, that exposes the `deliverator` binary. The binary should support:
- `deliverator start`
- `deliverator open`
- `deliverator logs`

The workspace should also expose a contributor-only Bun watch command, `bun run dev`, that reruns the local runtime on source changes. This is a development affordance, not part of the end-user product contract.

`deliverator start` starts the Fastify server in the foreground and uses port `0` by default so the operating system assigns a free port. After Fastify binds, the CLI writes `pid`, `port`, `url`, and `startedAt` into `~/.deliverator/run/current.json` and prints the URL to stdout. This command does not daemonize the process.

`deliverator open` reads `~/.deliverator/run/current.json`, checks that the stored URL is reachable, prints the URL, and opens it with a cross-platform browser opener.

`deliverator logs` searches JSONL logs. It should support simple local filters for `project`, `task`, `run`, substring grep, and follow mode. It reads files directly from `~/.deliverator/logs` and `<project>/.deliverator/local/logs`.

### Bun-backed development flow

The workspace root should switch to Bun for dependency installation and script execution:
- `bun install`
- `bun run dev`
- `bun run typecheck`
- `bun run lint`
- `bun run test`

`bun run dev` should start the same application runtime in watch mode so contributor feedback remains fast without bringing back Docker or a separate process manager. The watch-mode contract does not need to preserve the exact same PID or port across restarts, but it must keep the local iteration loop short and must continue to print the active URL after each restart.

### Startup

`apps/server/src/main.ts` should boot the global application config and registry storage under `~/.deliverator` first. In development, if the registry is empty, the bootstrap path may register the current workspace as a sample project so the app remains usable on first run. Project-local databases should not be opened eagerly for every route at startup unless the registry is tiny and that keeps the code simpler; lazy context resolution is acceptable.

### HTTP routes

Global routes:
- `/dashboard`
- `/feed`
- `/projects`
- `/api/projects`
- `/api/dashboard`
- `/api/feed`
- `/api/events/stream`

Project-scoped routes:
- `/projects/:projectSlug/board`
- `/projects/:projectSlug/tasks/:taskId`
- `/api/projects/:projectSlug/config/compiled`
- `/api/projects/:projectSlug/board`
- `/api/projects/:projectSlug/board/schema`
- `/api/projects/:projectSlug/tasks/:taskId`

`/` redirects to `/projects` when the registry is empty and otherwise redirects to the last selected project board.

### SSE

Global SSE remains one stream, but payloads must include `projectSlug` so the client can invalidate project-scoped queries correctly and global feed entries can render source-project identity.

## UI Model

The app shell gains a project switcher. Dashboard and feed remain app-global operator views. Board and task detail become project-scoped surfaces, and their query keys must include `projectSlug`.

The Projects screen stops being a placeholder and becomes the place to:
- list registered projects
- add a project by filesystem path
- show path/health metadata
- navigate to the project board

Task identity is no longer globally unique from the client’s perspective. The UI must always carry `projectSlug` alongside `taskId`.

## Logging and Local Diagnostics

The current `pino` logger already emits structured logs. Keep that approach and standardize on JSONL files as the local diagnostic layer. Every log line should remain machine-readable and should continue to include stable keys such as `service`, `correlation_id`, `trace_id` when available, `task_id`, `run_id`, and `action_run_id`.

The design explicitly drops:
- OpenTelemetry export
- Prometheus scraping
- Grafana dashboards
- Loki/Promtail shipping

This phase does not replace those systems with another backend. It replaces them with searchable local JSONL logs and CLI-level inspection.

## Documentation Split

`README.md` should become a product-facing document: what DELIVERATOR is, what problem it solves, how the per-project model works at a high level, and why the product is useful. It should not lead with contributor setup, package-manager activation, or local runtime internals.

Contributor setup, Bun commands, watch-mode workflow, validation commands, and local troubleshooting should move into `DEVELOPMENT.md`. `CONTRIBUTING.md` should remain focused on contribution flow, planning expectations, testing/review expectations, and links to `DEVELOPMENT.md` for environment setup.

## Git Ignore Management

On project registration, DELIVERATOR should inspect the target path. If it is a git repository or has a writable `.gitignore`, append an ignore rule for `.deliverator/local/` if missing. Do not add a rule for `.deliverator/` as a whole because `shared/` is supposed to remain committable. If the path is not a git repository, registration succeeds and returns a warning that ignore management was skipped.

## Failure Modes and Recovery

- If a project path does not exist, registration fails with a validation error before creating registry records.
- If `.deliverator/shared` exists with customized files, initialization must not overwrite them.
- If `.deliverator/local` exists, migrations should rerun safely using the existing migration-tracking table.
- If one project database is unavailable during global dashboard/feed fan-out, the app should return a degraded-but-explicit response rather than hiding the failure.
- If the DELIVERATOR app repo is also a managed project, the project-local `.deliverator/` in that repo must remain separate from the global app state under `~/.deliverator`.
- If `deliverator open` finds stale runtime metadata, it should fail clearly and instruct the operator to run `deliverator start` again.
- If a JSONL log line is malformed, `deliverator logs` should skip it with a warning rather than aborting the whole query.

## Testing and Review Expectations

Tests must prove:
- project registration creates the expected filesystem layout
- `.gitignore` behavior is correct for git and non-git targets
- project databases remain isolated
- global dashboard/feed aggregate across multiple projects
- project board/task routes never leak tasks from other projects
- the root redirect behavior changes based on whether registered projects exist
- `deliverator start` writes `~/.deliverator/run/current.json` after binding
- `deliverator open` reads that runtime metadata and resolves the current URL
- `deliverator logs` can find JSONL entries by simple filters without Docker or an external backend
- `bun run dev` restarts the local runtime on source changes or equivalent watch triggers

Code review must confirm:
- project state is no longer stored in the app registry database
- global registry and app-state data live under `~/.deliverator`, not the DELIVERATOR repo working tree
- Docker and the local telemetry stack are no longer part of the supported runtime contract
- Bun, not `pnpm`, is the documented and wired package-manager/command layer
- board/task logic is project-scoped at both API and UI layers
- dashboard/feed are intentionally global
- `.deliverator/shared` vs `.deliverator/local` semantics are consistently documented and enforced
- the root `README.md` is product-facing and contributor setup has moved to `DEVELOPMENT.md`
